# 教訓:擴展高寫入時序攝取路徑

## 目的

一份去識別化的設計案例,來自在生產環境營運**高寫入時序攝取路徑**(裝置/感測器串流持續落地)。這是我投入最多設計心力的部分,因此在此以代表性數字與每個決策背後的取捨完整記錄。

這是**教訓敘事,不是 kit 功能**。`ModulithReliabilityKit` 骨架並未提供時序模組;本文件所有可對程式查證的宣稱都留在 `src/`。此文捕捉問題的*工程形狀*與推理——與 kit 展示的「真實負載下的可靠性」同一命題,只是下沉到儲存/寫入層。內容**不含任何專有識別資訊**;規模數字為取整後的代表性量級(見 [README](../../README.md#scope-provenance-and-ai-assisted-authorship) 的去識別化聲明)。

## 情境(工作負載的形狀)

- 裝置遙測的連續串流:每台裝置每幾秒回報一次位置/狀態。
- 兩種寫入形狀並存:
  1. **僅追加的歷史**表(每筆讀數保留一段有界視窗),與
  2. 小型的**「每裝置最新狀態」**表(每裝置一列,持續被覆寫)。
- 量級:**每日數億次寫入**,追加歷史的攝取約 **~100+ GB/日(未壓縮)**。

## 問題 1 —— 寫入放大拖垮「最新狀態」表

最新狀態列在**每一筆**進來的讀數都被 upsert:

```sql
INSERT ... ON CONFLICT (device_key) DO UPDATE SET ...   -- 每個遙測封包都跑
```

在 PostgreSQL **MVCC** 下,`UPDATE` 不是原地修改:它把舊版本標記為死元組,再插入新列。於是每封包一次無條件 upsert 導致:

- **~100% 的 update ratio**(幾乎每次觸碰都是 update 而非 insert),且
- **死元組攀升超過表的 ~80%**,快過 autovacuum 的回收速度——拖慢查詢延遲,並使表/索引膨脹。

**根因:** 這不只是維運/vacuum 問題,而是**程式碼 + 架構**問題:無條件寫入,且把即時熱狀態在封包路徑上直接寫進持久儲存。

## 分層解法(依「投入 vs 回報」排序)

| 解法 | 概念 | 寫入降幅 | 難度 |
| ---- | ---- | -------- | ---- |
| **條件更新** | 只在*實質*變化時才 upsert——位置移動超過門檻、速度變化超過門檻,或超過有界「心跳」間隔 | 少 ~60–80% 更新 | 低(純 SQL) |
| **HOT + fillfactor** | 降低 `fillfactor`(如 70)並避免更新索引欄,讓 PostgreSQL 走 Heap-Only-Tuple 更新、省去索引異動 | 少寫索引;與上者互補 | 低(`ALTER TABLE`) |
| **write-behind 快取** | 熱「最新狀態」留在記憶體;以有界間隔(如每 ~30 秒)刷入持久表,而非每封包 | 少至 ~97% 更新 | 中(架構) |
| **continuous aggregates** | 讓時序擴充以刷新策略從歷史表維護一個物化的「最新」視圖,取代手工維護的可變表 | 寫入變成擴充的問題 | 中 |

便宜解法的關鍵,正是把 kit 的冪等直覺套到寫入上:**沒有改變任何東西的寫入,就不該發生。** 在 `ON CONFLICT DO UPDATE` 的 `WHERE` 放一個「移動/門檻」判斷,就把「每封包都寫」變成「只在真的變化時才寫」——既正確又便宜得多。

## 問題 2 —— 在不破產的前提下保留 ~100 GB/日 的歷史

僅追加歷史保留一段有界視窗,但量仍巨大。兩個槓桿:

- **欄式壓縮**(TimescaleDB 式 hypertable 壓縮),以 `device_key` 分段、`timestamp DESC` 排序。這種資料形狀的壓縮比極端——**~99% 以上**(數 GB 的 chunk 縮到數十/數百 KB),因為每裝置、時間排序的數值/位置資料壓得極好。延遲壓縮(如 24–72 小時後),讓仍在變動的近期視窗保持未壓縮而快速。
- **有界保留**:自動丟棄視窗外的 chunk,而非跑一次巨大的 `DELETE`。

### 分層儲存 —— 以及埋伏其中的鎖

用**熱/溫/冷**生命週期調校成本/效能:

| 層級 | 介質 | 年齡 | 狀態 |
| ---- | ---- | ---- | ---- |
| 熱 | SSD | 近數日 | 未壓縮、快 |
| 溫 | HDD | 數週–數月 | 已壓縮(~200:1) |
| 冷 | 物件儲存(S3) | 歸檔 | Parquet(~300:1) |

隱蔽的 bug:第一版用 `ALTER TABLE ... SET TABLESPACE` 在 tablespace 間搬 chunk,而這會取得 **`ACCESS EXCLUSIVE` 鎖**,在搬動整個數 GB chunk 期間阻塞生產查詢。重設計讓維護**感知鎖且非阻塞**:

1. **階段 A** —— 原地壓縮(近乎非阻塞,依擴充自身排程)。
2. **階段 B** —— 只搬**已壓縮**的 chunk(數十 MB,而非數 GB),一次一個,chunk 間留小延遲,設時間預算,於離峰執行。

搬 25 MB 而非 5 GB、分散且設時間盒,把每 chunk 的停頓從「數秒全域阻塞」降到亞秒級。**教訓:在規模下,維護作業的鎖特性,和它所維護的功能一樣重要。**

## 問題 3 —— 十億列規模下的列寬

當列數達數億到數十億,*列的寬度*是一等成本:

- **鍵型別**:`UUID` 鍵每個 16 bytes;`BIGINT` 代理鍵是 8。兩個 UUID 欄 → 32 bytes/列;改 `BIGINT` 大致砍半,並縮小每個帶鍵的索引(每十億列約省 ~15 GB,索引再小 30–40%)。
- **欄位編碼**:以文字存的狀態(`"10000001"`)若其實是 bitmask,該放 `SMALLINT`。每列的小節省在此列數下累積成 GB。

這些是「無聊」的勝利,但在十億列時,帳單常是列寬而非查詢。

## 這如何連到 kit

- **冪等寫入** —— 條件更新判斷,是 kit 冪等 inbox(`ON CONFLICT DO NOTHING` 攝取)在寫入路徑的手足:不套用沒有新資訊的效果。
- **有界、顯式的工作單元** —— 以排程刷入熱狀態而非在封包路徑上,與 kit「具決定性資源生命週期的專屬背景 drain」同一直覺。
- **可維運性是設計輸入** —— 一如 kit 把 retry/dead-letter/reprocess 當一等公民,這裡的*維護*作業(壓縮、分層)是按其鎖與成本特性設計,而非事後硬加。

## 值得抄

1. **讓寫入有條件。** no-op 寫入仍是一次 MVCC 寫入;用「真變化」判斷守住 upsert。
2. 事件率高時,**把熱可變狀態移出每事件的持久路徑**——以有界間隔 write-behind。
3. **以策略壓縮 + 保留**,而非手刻 DELETE;壓縮的分段/排序對齊你的主查詢鍵。
4. **為維護的鎖特性做設計。** 寧可搬小的已壓縮 chunk、設時間盒且分散,也別做大的阻塞操作。
5. 進入數億列régime 後,**把列寬當成本**看待。

## 別盲目抄

- 精確門檻(距離/速度/心跳)、壓縮延遲、分層年齡都**因工作負載而異**——量測你自己的。
- write-behind 快取是**用熱副本的持久性換寫入量**;只有在持久歷史是真相來源、且「最新狀態」可重建時才可接受。
- 分層儲存帶來真實的維運面(tablespace、物件儲存歸檔、還原路徑)。在單層壓縮儲存真的成為瓶頸前,別採用。
